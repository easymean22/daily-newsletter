# #6 UI에 미치는 결정사항 (사용자 답변 #6 반영)

**요일당 태그는 다중 선택을 허용한다.** 즉 `SettingsEntity`의 요일별 태그 키는 **단일 문자열이 아니라 콤마(또는 JSON 배열) 직렬화된 N개**를 저장한다 (구체 직렬화 형태는 #6 ADR/플랜에서).

함의:

1. **데이터 모델**: 요일 → `List<String> tagNames`. 빈 리스트는 "이 요일은 태그 필터를 걸지 않는다"로 해석 가능 (의미적으로 "전부 매칭" 또는 "off" — #6에서 결정).
2. **매칭 알고리즘**: 다중 태그 선택 시 매칭이 **OR**(어느 하나라도 일치하면 후보)인지 **AND**(전부 일치해야 후보)인지는 **#6 핸드오프에서 결정**해야 한다. 본 플랜의 권고는 OR (lazy 매칭 후보 풀이 넓어져 0건 엣지를 줄임) — 다만 결정권은 #6에 둔다.
3. **lazy 생성 트리거**: 매칭 0건 판정도 OR/AND 정책에 따라 달라진다 (#7 영역).
4. **본 플랜의 Repository 시그니처**: `findUnprintedNewsletterByTag(tagName: String)`은 단일 태그를 받는다. 다중 태그를 받는 변형(`findUnprintedNewsletterByAnyTag(tagNames: List<String>)` 등)은 #6/#4가 도입할 때 추가. 본 플랜은 단일 진입점만 책임.

# UI 영향 (요약 — 본 플랜에서 디자인하지 않음)

- **키워드 입력 UI**(핸드오프 #1): 등록 폼에 태그 칩 입력 필드 (사용자가 직접 입력).
- **주제 작성 UI**(핸드오프 #2): 직접 작성 경로 + 자동 생성 경로 모두 사용자가 칩으로 태그를 직접 입력한다. **자동 생성 경로의 Claude 응답은 태그를 포함하지 않는다** (라운드 1). 사용자가 태그를 비워둬도 invariant에 의해 `모든주제`가 자동 부착됨을 UI 사본/안내에 반영할지는 #2 결정.
- **뉴스레터 표시**(영향 적음): 카드에 태그 배지를 보일지 여부는 #6 또는 별도 결정.
- **요일별 설정 UI**(핸드오프 #6): 태그 셀렉터의 옵션 풀 = Newsletters DB의 multi_select 옵션 + 시드 `모든주제`의 합집합. 정규화 키로 dedupe. **다중 선택 가능한 칩/체크박스 UI**. (라운드 1: MVP 검증은 단일 슬롯 = `모든주제` 1개 슬롯에 한정. 다중 슬롯은 공개 배포 이월.)

본 플랜은 위 UI에 들어갈 **데이터 시그니처**만 책임. 화면 디자인은 각 핸드오프 플랜에서.

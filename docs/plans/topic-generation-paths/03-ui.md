# UI 설계

## 대안 (3개 비교)

### 옵션 U-A: 모든 트리거를 한 화면에 (Topics 화면)

- Topics 화면 상단에 "주제 생성" 버튼 + "직접 작성" 입력 폼 토글.
- 키워드 화면은 그대로 (자동 경로는 BG에서 트리거).
- 장점: 사용자가 "주제는 여기서 만든다"는 단일 멘탈 모델.
- 단점: 자동 경로(키워드 추가 직후)의 가시성이 낮음 → "방금 추가한 게 뭐했나" 의문.

### 옵션 U-B: 트리거 위치를 경로별로 분산

- (a) 자동: 키워드 추가 시 BottomSheet 닫히면서 스낵바 ("주제 N개 생성 중…" → "완료/실패").
- (b) 수동 버튼: Topics 화면 상단 FAB 또는 TopAppBar 액션.
- (c) 직접 작성: Topics 화면에서 별도 진입점 (FAB 메뉴 분리 또는 TopAppBar의 "추가" 액션 → BottomSheet).
- 장점: 각 트리거가 사용자가 그 행동을 하는 자연스러운 맥락에 위치.
- 단점: (b)와 (c) 두 액션이 같은 화면에 있어 변별이 필요.

### 옵션 U-C: 모달 전체화면 + 경로 선택

- "+ 주제 추가" 1개 진입점 → 전체화면 모달에서 "AI에게 맡기기 / 직접 쓰기" 선택.
- 장점: 진입점 단일.
- 단점: 자동 경로가 사라짐 (키워드 등록 시 즉시 트리거가 spec 의도). 사용자 선택 단계가 늘어 마찰.

## Trade-off (의도 부합도 > 단순성 > 접근성 > 입력 부담 > 오류 복구)

| 축 | U-A (Topics 1화면 통합) | U-B (경로별 분산) | U-C (모달 + 경로 선택) |
| --- | --- | --- | --- |
| **spec 의도 부합** | 자동 경로의 즉시성 표현 약함. | spec §3의 "사용자 행동이 트리거" 흐름과 1:1 매핑. | 자동 경로(키워드 트리거) 부재. **즉시 기각.** |
| **사용자 단계 수** | 적음 (단일 화면). | 적음 (각 트리거가 맥락에 있어 한 번에 발동). | 많음 (+ 모드 선택 단계). |
| **자동 경로 가시성** | 낮음 — 다른 화면에서 도는 일이 안 보임. | 높음 — 추가 직후 같은 화면 스낵바. | — |
| **변별성 ((b) vs (c))** | 동일 화면 + 동일 진입 → 혼동. | TopAppBar 액션 vs FAB 같이 명시적 분리 가능. | 모드 선택 단계로 명확. |
| **입력 부담 ((c) 경로)** | BottomSheet — 중간. | BottomSheet — 중간. | 전체화면 — 큼 (overkill). |
| **오류 복구** | 한 화면에서 모두 — 좋음. | 각 화면에서 — 좋음. | 모달 닫고 다시 — 약함. |

## 추천안: **옵션 U-B (경로별 분산)**

근거:
1. **spec §3 의도와 1:1 매칭** — 키워드 추가는 키워드 화면, 주제 관리는 주제 화면.
2. **자동 경로가 사용자에게 보임** — 키워드 추가 직후 스낵바가 즉시 피드백.
3. **(b)와 (c)의 변별을 명확히 가능** — TopAppBar의 Refresh(↻) 아이콘 = 수동 생성 (현재도 있음, 의미 재정의), FAB(+) = 직접 작성.
4. **MVP의 "잠깐 쓰는 도구" 디폴트와 일치** — 화면/선택지 최소.

## 와이어프레임 (텍스트 스케치)

### KeywordScreen — 자동 경로 트리거 (a)

```
[TopAppBar] 키워드 / 메모             [필터]
─────────────────────────────────────────────
LazyColumn (기존 카드 목록)
─────────────────────────────────────────────
                              [+ FAB] (기존)

# BottomSheet (기존 + 태그 입력 추가)
┌─────────────────────────────────┐
│ 새 키워드 / 메모                │
│ [내용 OutlinedTextField]        │
│ [타입 Dropdown: 키워드/메모]    │
│ [태그 (선택) ChipInput]    NEW │  ← 0~N개 칩
│           [취소] [저장]         │
└─────────────────────────────────┘

# 저장 직후
─────────────────────────────────────────────
Snackbar: "키워드 추가 완료. 주제 생성 중…"
  ↓ (자동 경로 완료 시)
Snackbar: "주제 N개가 생성됐어요" [보기]  → Topics 화면으로 이동
  또는
Snackbar: "주제 생성에 실패했어요. 주제 화면에서 다시 시도해주세요"
```

태그 입력 컴포넌트는 단순 칩 입력 (+ 버튼 누르면 텍스트 필드 → Enter/Done으로 칩 생성). 옵션 풀 자동완성은 본 플랜 scope 밖 (전제 플랜 5단계 헬퍼 도입 시 후속 #6에서 자연 활용).

### TopicsScreen — 수동 버튼 (b) + 직접 작성 (c) 트리거

```
[TopAppBar] 오늘의 주제          [↻ 생성] [+ 직접]
─────────────────────────────────────────────
LazyColumn (기존 카드 목록)

(빈 상태) "아직 선정된 주제가 없습니다.
          상단 ↻ 또는 + 버튼으로 추가하세요."
─────────────────────────────────────────────

# (b) [↻ 생성] 탭 시 — 즉시 실행
1. 로딩 인디케이터 (현재의 isLoading 재사용)
2. 완료 시 Snackbar "주제 N개 추가됨"
3. 실패 시 Snackbar "생성 실패: <메시지>"

# (c) [+ 직접] 탭 시 — BottomSheet
┌─────────────────────────────────┐
│ 직접 주제 작성                  │
│ [제목 OutlinedTextField]        │
│ [태그 (선택) ChipInput]         │
│           [취소] [저장]         │
└─────────────────────────────────┘
저장 → 즉시 Topics 목록에 반영.
```

(b)의 아이콘은 기존 `Icons.Default.Refresh`를 그대로 재사용하되 의미가 "재생성"에서 "수동 생성"으로 미세 변경. 빈 상태 안내 문구로 의미 보강.

### Topics 카드 — 태그 배지 추가 (선택적)

```
┌─────────────────────────────────┐
│ <주제 제목>          [편집] [삭제] │
│ [직접 지정] [#IT] [#모든주제]    │
│ 관련: <키워드들>                │
└─────────────────────────────────┘
```

근거: 자동 경로에서 Claude가 부여한 태그를 사용자가 즉시 확인할 수 있어야 (b) 재생성 또는 (c) 직접 작성으로 보정할지 판단 가능. invariant 덕분에 항상 최소 `모든주제` 1개 배지가 보임.

## 컴포넌트 트리 / 상태 흐름

```
KeywordScreen
 └─ KeywordViewModel
     ├─ KeywordRepository.addKeyword(text, type, tags) → KeywordUiItem
     ├─ ClaudeTopicSuggester.suggest(...)  ← 자동 경로
     ├─ TopicRepository.saveTopic(...) × N  ← 자동 경로 결과 저장
     └─ NewsletterRepository.listAvailableTagNames() (선택)

TopicsScreen
 └─ TopicsViewModel
     ├─ TopicRepository.getTodayTopics()
     ├─ TopicRepository.deleteTodayTopics()    ← regenerate에서 호출
     ├─ ClaudeTopicSuggester.suggest(...)      ← (b) 수동 버튼
     ├─ TopicRepository.saveTopic(...) × N
     └─ TopicRepository.saveTopic(title, "direct", emptyList(), tags)  ← (c) 직접 작성
                                                priorityType은 "direct" 고정
```

새 ViewModel 의존성:
- `KeywordViewModel`: + `ClaudeTopicSuggester`, `TopicRepository`, `NewsletterRepository`(선택).
- `TopicsViewModel`: + `ClaudeTopicSuggester`, `KeywordRepository`, `NewsletterRepository`(선택).

## 에러 / 빈 상태

| 상황 | UI 처리 |
| --- | --- |
| 자동 경로: API Key 미설정 | Snackbar "API 키를 먼저 설정해주세요" + [설정으로] 액션. 키워드 추가는 성공. |
| 자동 경로: Claude 호출 실패 | Snackbar "주제 생성에 실패했어요. 주제 화면에서 다시 시도해주세요". |
| 자동 경로: Claude 응답 0개 또는 파싱 실패 | Snackbar "이번엔 새 주제가 만들어지지 않았어요". |
| (b) 수동 버튼: pending 키워드 0개 | 버튼 enabled 유지(탭 시 안내). 탭 시 다이얼로그 "키워드/메모를 먼저 추가해주세요" + [키워드로] 액션. (또는 disabled — 사용자 확인 필요 #3) |
| (b) 수동 버튼: Claude 호출 중 사용자가 다시 탭 | 첫 호출 진행 중이면 두 번째 탭 무시 (isLoading guard). |
| (c) 직접 작성: 제목 빈 입력 | 저장 버튼 disabled. |
| (c) 직접 작성: 태그에 쉼표 입력 | 인라인 에러 "태그에 쉼표를 포함할 수 없어요". |
| Topics 빈 상태 | "아직 선정된 주제가 없습니다. 상단 ↻ 또는 + 버튼으로 추가하세요." |
| Topics 로딩 | 기존 CircularProgressIndicator 재사용. |

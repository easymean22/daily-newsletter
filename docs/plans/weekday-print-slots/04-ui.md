# UI 설계

## UI 구조 대안 비교

### 옵션 U-A: `SettingsScreen`에 "요일별 프린트" 진입점 Card → 별도 `WeeklyScheduleScreen` 화면 (추천)

- 설정 화면의 "프린트 시간" 기존 Card를 "요일별 프린트 설정" Card로 교체. 탭 시 별도 Navigation route로 이동.
- `WeeklyScheduleScreen`: 요일 7개 카드 세로 나열. 각 카드 내 시각·on/off·슬롯 리스트.
- 슬롯 편집은 BottomSheet.

### 옵션 U-B: `SettingsScreen` 내부에 요일 7개 카드 섹션 직접 삽입 (인라인)

- 설정 화면이 길어짐 (API 키 + 프린터 + 요일 7개 + 기타).
- Navigation 변경 없음.

### 옵션 U-C: Bottom Navigation에 "스케줄" 탭 추가 — `ui/schedule/`을 1급 화면으로 승격

- 메인 네비게이션 레벨에서 진입.
- 사용자가 자주 수정하는 화면이 아니라 과장.

## Trade-off

| 축 | U-A (설정 진입점 + 별도 화면) | U-B (설정 화면 인라인) | U-C (Bottom Nav 탭) |
|---|---|---|---|
| **spec 의도 부합** | "설정 UI 일부"로 spec §3이 분류 — 별도 화면도 일부로 해석 가능. | 동일. | spec이 1급 탭을 요구하지 않음 — 과확장. |
| **사용자 멘탈 모델** | "설정에서 → 스케줄" 2-hop. 익숙한 pattern. | "설정이 길다" — 스크롤 피로. | "스케줄이 주 기능"처럼 보임 (실제 주 기능은 뉴스레터). |
| **입력 부담** | 요일별 편집이 독립 화면이라 집중 가능. 뒤로가기 1회로 설정 복귀. | 동일 화면에 섞여 편집 context 전환 비용. | 진입 비용 낮으나 오용 가능성. |
| **구현 비용** | 중 — 1 Composable + 1 ViewModel + Navigation route 1개. | 낮 — SettingsScreen만 수정. | 중+ — Navigation 구조 수정. |
| **미래 확장 (요일별 프린터 등)** | 자연스러움 — WeeklyScheduleScreen 내부 확장. | 설정 화면 더 길어짐. | 탭 내부 sub-navigation까지 가능. |
| **접근성 (스크롤/검색)** | 요일 카드만 보이는 전용 화면. | 설정 항목 속에 묻힘. | — |

## 추천안: **옵션 U-A**

근거:

1. **spec이 "설정 UI"라고 분류 — 메인 네비 탭 승격은 과확장**.
2. **입력 부담이 가장 낮음** — 요일 7개 집중 편집 화면.
3. **다른 플랜과의 영향 최소** — newsletter-shelf의 `NewsletterScreen` 수동 생성 FAB 추가 작업과 충돌 없음.
4. **Navigation 추가 1개** — 기존 패턴(`ui/<feature>/<Feature>Screen.kt` + `<Feature>ViewModel.kt`) 유지.

### Slot 편집 BottomSheet

```
┌─────────────────────────────────┐
│ 월요일 슬롯 추가                  │
│                                  │
│ 태그     [드롭다운 ▼]            │
│   (모든주제, IT, 경제, …)        │
│   ※ 이미 월요일에 있는 태그는    │
│     비활성                       │
│                                  │
│ 장수     [-] 2 [+]   (1~5)       │
│                                  │
│           [삭제]  [취소] [저장] │
└─────────────────────────────────┘
```

- 편집 모드일 때 "삭제" 버튼 노출. 추가 모드는 감춤.
- 드롭다운 소스: `NewsletterRepository.listAvailableTagNames()` (tag-system 5단계 이월).
- 이미 같은 요일에 등록된 태그는 disabled 표시 + "이미 추가됨" 부제. **현재 편집 중인 슬롯 본인 태그는 disabled 제외** (자기 자신은 재선택 가능).

## 컴포넌트 트리 / 상태 흐름

```
WeeklyScheduleScreen
 └─ WeeklyScheduleViewModel
     ├─ SettingsRepository.observeWeeklySchedule(): Flow<WeeklySchedule>
     ├─ SettingsRepository.setWeeklySchedule(...)
     ├─ NewsletterRepository.listAvailableTagNames(): List<String>   // 드롭다운 소스
     └─ WorkScheduler.scheduleAll()                                   // 저장 후 재schedule
```

### UI 상태 모델

```kotlin
data class WeeklyScheduleUiState(
    val schedule: WeeklySchedule = WeeklySchedule.default(),
    val availableTagNames: List<String> = emptyList(),
    val isLoading: Boolean = false,
    val editingSlot: SlotEditTarget? = null,    // BottomSheet 상태
    val editingTime: DayOfWeek? = null,         // 시각 편집 Dialog 상태
    val error: String? = null,                  // 저장 실패, 파싱 실패 등
    val saveStatus: SaveStatus = SaveStatus.Idle
)

data class SlotEditTarget(
    val day: DayOfWeek,
    val slotIndex: Int?,    // null이면 신규
    val currentTag: String = "",
    val currentPages: Int = 2,
    val validation: ValidationResult = ValidationResult.Ok
)

sealed class SaveStatus {
    object Idle : SaveStatus()
    object Saving : SaveStatus()
    object Saved : SaveStatus()                  // 스낵바 "저장됐어요"
    data class Failed(val message: String) : SaveStatus()
}
```

### ViewModel 주요 메서드

```kotlin
@HiltViewModel
class WeeklyScheduleViewModel @Inject constructor(
    private val settingsRepository: SettingsRepository,
    private val newsletterRepository: NewsletterRepository,
    private val workScheduler: WorkScheduler
) : ViewModel() {

    val uiState: StateFlow<WeeklyScheduleUiState>

    fun toggleDay(day: DayOfWeek, enabled: Boolean) { ... }          // OFF 복귀 자동 방지는 저장 스키마가 해결
    fun setDayTime(day: DayOfWeek, hour: Int, minute: Int) { ... }

    fun openSlotEdit(day: DayOfWeek, slotIndex: Int?) { ... }
    fun updateEditingSlotTag(tag: String) { ... }                    // validation 즉시 갱신
    fun updateEditingSlotPages(pages: Int) { ... }
    fun saveEditingSlot() {
        // validation → 성공 시 schedule 업데이트 → setWeeklySchedule → workScheduler.scheduleAll()
    }
    fun deleteEditingSlot() { ... }
    fun dismissSlotEdit() { ... }

    fun openTimeEdit(day: DayOfWeek) { ... }
    fun dismissTimeEdit() { ... }
}
```

### 저장 흐름 (설정 변경 시마다)

1. 사용자 액션 → ViewModel의 mutator (toggleDay, saveEditingSlot 등).
2. ViewModel이 새 `WeeklySchedule` 계산.
3. `settingsRepository.setWeeklySchedule(new)` 호출.
4. `workScheduler.scheduleAll()` 호출 (성공 시 `SaveStatus.Saved`).
5. UI는 `observeWeeklySchedule()` Flow를 통해 자동 반영.

실패 시 `SaveStatus.Failed(message)` — 스낵바로 피드백.

## 에러 / 빈 상태

| 상황 | UI 처리 |
|---|---|
| **태그 유일성 위반 (Q4 추천 B: 드롭다운에서 이미 쓰인 태그 disabled)** | 드롭다운 항목에 "(이미 추가됨)" 부제 + 선택 불가. 편집 중인 슬롯 본인 태그는 제외. |
| 태그 옵션 풀이 비어 있음 | 드롭다운에 "태그가 없습니다. 주제를 먼저 추가해주세요" + 드롭다운 disabled + [주제로 이동] 버튼. |
| 장수 범위 벗어남 (<1 or >5) | Stepper의 - / + 버튼이 경계에서 disabled. 입력 자체 차단. |
| 태그 공백만 | 저장 버튼 disabled. (드롭다운이 공백 선택을 차단하므로 실제로는 발생 안 함.) |
| 슬롯 0개 + enabled=true | 요일 카드에 "슬롯을 1개 이상 추가해주세요" 안내 + [+ 슬롯 추가] CTA. |
| 요일 전체 OFF | SettingsScreen 진입점 요약에 "활성 요일: 없음" 표시. |
| 저장 실패 (Room write 실패) | 스낵바 "저장 실패: <메시지>". UI 상태는 이전 값 유지 (낙관적 업데이트 롤백). |
| 스케줄 반영 실패 (WorkManager 예외) | 스낵바 "저장됐으나 스케줄 반영 실패 — 앱을 재시작하면 자동 복구됩니다". |
| 스키마 파싱 실패 (손상된 JSON) | 초기 로드 시 `WeeklySchedule.default()` 복귀 + 스낵바 "설정이 손상되어 초기화했습니다". |
| 시각 충돌 (같은 요일·같은 시각 다른 슬롯) | 허용 (슬롯은 같은 시각에 순차 처리 — spec §3). |
| 마이그레이션 진행 중 (앱 첫 기동) | Progress indicator 잠깐 노출. 실패 시 `default()` 복귀 + 재시도 유도. |

## 컴포넌트 재사용

- `OutlinedTextField` — 현재 사용.
- `TimePicker` + `AlertDialog` — `SettingsScreen`의 기존 시각 편집 패턴 재사용.
- `Card` — `SettingsScreen` 진입점 스타일.
- `ModalBottomSheet` + `Stepper` 패턴 — newsletter-shelf 03-ui.md의 Slot 편집 BottomSheet와 동일 UX. 공통 컴포넌트로 추출 여지 있으나 MVP 범위 밖 (동일 문구 두 군데 복사 1회 감수).

## 네비게이션

```
AppNavHost
 ├─ "home" → TopicsScreen 또는 NewsletterScreen
 ├─ "keywords" → KeywordScreen
 ├─ "newsletters" → NewsletterScreen
 ├─ "settings" → SettingsScreen
 └─ "settings/weekly-schedule" → WeeklyScheduleScreen    ← 신규
```

- `SettingsScreen`의 "프린트 스케줄" Card가 `navController.navigate("settings/weekly-schedule")` 호출.
- TopAppBar의 back button이 `SettingsScreen`으로 복귀.

## 접근성 / 반응성

- 요일 카드 토글: `Switch` 컴포넌트 — 터치 영역 48dp 이상.
- 시각 편집 Card — 전체 탭 영역, onClick 바인딩.
- 슬롯 편집 BottomSheet — `ModalBottomSheet` + `rememberModalBottomSheetState()`. 스크림 터치로 닫힘.
- 큰 폰트 대응: Card 내 Row가 Column으로 wrap되도록 `Arrangement.SpaceBetween` + `weight` 사용.

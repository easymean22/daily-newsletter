# UI 영향

spec §3은 "미프린트 Newsletter 집합"를 개념적 명칭으로만 쓰며 **별도 미프린트 Newsletter 집합 화면을 요구하지 않는다**. 기존 `NewsletterScreen`이 Newsletters DB 전체를 보여주므로 미프린트 Newsletter 집합 = 해당 화면의 `generated` 상태 항목.

## 변경 대안 (3개 비교)

### 옵션 U-A: 기존 NewsletterScreen에 "+ 수동 생성" 추가, 상태 문구 조정

- TopAppBar에 `+` 액션 추가 — 탭 시 태그·장수 선택 BottomSheet → `generateForSlot` 호출 → 미프린트 Newsletter 집합에 추가.
- 카드 상태 뱃지 문구 조정: "생성됨" → "미프린트 Newsletter 집합" (status가 `generated`일 때).
- 별도 탭/필터 추가하지 않음.

### 옵션 U-B: 탭으로 미프린트 Newsletter 집합 / 프린트 이력 분리

- 상단에 "미프린트(N) / 프린트 이력" 탭.
- 미프린트 Newsletter 집합 탭이 수동 생성 버튼을 포함.
- 정보 밀도 증가. 사용자가 탭 전환 학습 필요.

### 옵션 U-C: 별도 "미프린트 Newsletter 집합 화면" 신설 + 기존 NewsletterScreen은 프린트 이력 전용

- Navigation에 `ShelfScreen` 신설.
- 구조 변경이 커 MVP 범위를 벗어남.

## Trade-off

| 축 | U-A (기존 화면 확장) | U-B (탭 분리) | U-C (별도 화면) |
| --- | --- | --- | --- |
| **spec 의도 부합** | spec §3이 "Newsletters DB 한 풀"로 정의 — 단일 화면이 의도 1:1. | 미프린트 Newsletter 집합/이력 분리는 spec에 없음 — 추측. | spec 의도 이상의 구조 변경. |
| **사용자 멘탈 모델** | "뉴스레터 화면 = 내 뉴스레터 전부" 단순. | "미프린트 Newsletter 집합가 별개"라는 개념 학습. | 네비게이션 계층 ↑. |
| **MVP 단순성** | 최소 — FAB 1개 + 문구 2개. | 중 — 탭 컴포저블 추가. | 대 — 스크린·ViewModel·Navigation 전부 추가. |
| **구현 비용** | 낮음. | 중. | 높음. |
| **미래 확장 (필터)** | 카드에 상태 뱃지로 충분. 필요 시 필터 칩 추가 가능. | 탭으로 굳으면 필터는 탭 안에서 다시. | — |

## 추천안 (UI): **옵션 U-A**

근거: spec은 미프린트 Newsletter 집합 화면을 요구하지 않음. MVP 1일 E2E의 "프린트 시점에 2개 생성 → 1 출력, 1 비축" 검증이면 충분 — 사용자는 Notion에서 비축 상태를 직접 확인 가능. UI 구조 변경은 공개 배포 마일스톤에서 재고.

## 컴포넌트 트리 / 상태 흐름

```
NewsletterScreen
 └─ NewsletterViewModel
     ├─ NewsletterRepository.getNewsletters()        ← 목록 로드
     ├─ NewsletterRepository.printNewsletter(id)     ← 상세에서 즉시 프린트 (기존)
     └─ composeManualNewsletter(tag, pageCount):
        └─ NewsletterGenerationService.generateForSlot(tag, pageCount)
           ├─ (내부) TopicRepository.findPendingTopicsByTag(tag)
           ├─ (내부) ClaudeApi.createMessage
           ├─ (내부) NewsletterRepository.saveNewsletter(..., tags, topicIds, pageCount)
           └─ (내부) TopicRepository.markTopicsConsumed(selectedIds)
```

**NewsletterViewModel 의존성 추가**:

- `NewsletterGenerationService` — 수동 생성.
- `NewsletterRepository.listAvailableTagNames()` (tag-system 5단계) — 드롭다운 옵션 출처.

**신규 상태**:

```kotlin
data class NewsletterUiState(
    val newsletters: List<NewsletterUiItem> = emptyList(),
    val isLoading: Boolean = false,
    val error: String? = null,
    val availableTagOptions: List<String> = emptyList(),  // 신규
    val manualGenStatus: ManualGenStatus = ManualGenStatus.Idle  // 신규
)

sealed class ManualGenStatus {
    object Idle : ManualGenStatus()
    object Running : ManualGenStatus()
    data class Success(val title: String) : ManualGenStatus()
    data class Failed(val message: String) : ManualGenStatus()
}
```

## 에러 / 빈 상태

| 상황 | UI 처리 |
| --- | --- |
| 수동 생성: 태그 선택 안 됨 | 생성 버튼 disabled. |
| 수동 생성: 태그 옵션 풀이 비어 있음 | 드롭다운에 "태그가 없습니다. 주제를 먼저 추가해주세요" + 드롭다운 disabled. |
| 수동 생성: Claude API Key 미설정 | 생성 시도 시 스낵바 "API 키를 먼저 설정해주세요" + [설정으로]. |
| 수동 생성: 주제 후보 0개 (선택 태그 기준) | 스낵바 "해당 태그의 pending 주제가 없습니다. 주제 화면에서 추가해주세요" + [주제로]. |
| 수동 생성 중 사용자가 다시 탭 | BottomSheet 자동 닫힘, isLoading guard. |
| 카드 상태 뱃지 문구 | `generated` → "미프린트 Newsletter 집합", `printed` → "프린트 완료", `failed` → "실패". |
| 목록 빈 상태 | "아직 뉴스레터가 없습니다. 프린트 시각이 되면 자동으로 채워지거나, 상단 +로 직접 생성할 수 있어요." |
| 목록 로딩 | 기존 CircularProgressIndicator 재사용. |

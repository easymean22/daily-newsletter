# MVP: 데일리 종이 뉴스레터 자동 수신

## 1. 성공의 한 줄 정의
사용자가 프린트 시각에 Canon G3910에서 그날의 뉴스레터 종이를 손에 쥐는 경험이 **한 번 끝까지 성공**하는 상태.

## 2. WHY — 왜 이걸 만드나
- 개인이 매일 큐레이션된 뉴스레터를 **종이로** 받아 스크린 피로 없이 소비하기 위해.
- 주제 선정·콘텐츠 생성·출력을 자동화해 사용자의 수동 개입을 최소화.
- 공개 배포는 별도 마일스톤. MVP는 본인 1명 기준.

## 3. MVP 범위 — WHAT 들어가야 하나

> ### 2026-04-19 라운드 1 확정 (상위 규칙)
>
> 아래 5가지는 §3 이하 모든 기술(記述)과 §8 핸드오프에 선행하는 **상위 규칙**이다. 본문과 모순이 있으면 이 블록이 우선.
>
> 1. **MVP 슬롯 구성은 단일 슬롯 1경로**로 한정. 요일당 슬롯 묶음은 `[(모든주제, N장)]` 하나. 다중 슬롯(요일 × 서로 다른 태그 여러 개) 격리·순서·매칭 검증은 공개 배포 마일스톤으로 이월.
> 2. **"진열대/Shelf"는 제품 개념이 아니다.** Newsletter 레코드 = 생성된 아티클 1건. 아직 프린트되지 않은 Newsletter 레코드의 집합에는 별도 공간명(진열대·shelf·풀)을 부여하지 않는다. 본문 어휘도 "미프린트 Newsletter 레코드" 같은 중립 표현을 쓴다.
> 3. **Claude는 태그 시스템에 관여하지 않는다.** 태그 자동 추천·자동 분류 모두 제거. 태그 생성과 Topic에의 태그 부여는 **전부 사용자 수동**이다. "새 태그 동의 프롬프트" 규칙은 불필요해져 폐기.
> 4. **`자유주제`는 `모든주제`로 rename.** 동일 개념이며, invariant는 유지 — 모든 Topic에 기본 부여되고, 사용자는 이 태그를 수정·삭제할 수 없다.
> 5. **Topic 생성 시 태그 배정 기본값은 `모든주제` 단독.** 사용자가 이후 수동으로 추가 태그를 붙이고 뗀다. 자동 추천·자동 분류·retroactive 분류는 전부 없다.
>
> 영향 범위(재작업 대상) — designer 핸드오프 #3, #4 전면 재진입. 관련 ADR/plan은 §8 뒤 핸드오프 선언에 따른다.

### 파이프라인 구동축 (중요한 재정의)
기존 코드는 T−2h/T−30m/T의 시간축에 묶여 있으나, MVP의 구동축은 **사용자 행동 + 프린트 시점**으로 재정의한다.

- **주제 생성은 사용자 행동이 트리거**한다. 키워드 입력 직후, 또는 수동 주제 생성 버튼, 또는 사용자가 직접 주제를 작성해 넣는 3가지 경로로 Topics DB에 주제가 쌓인다.
- **뉴스레터는 사용자가 평소에 수동으로 만들어 Notion Newsletters DB에 미프린트 상태로 쌓아 둔다.** 프린트 시점엔 Newsletters DB에서 매칭되는 미프린트 레코드를 꺼내 쓴다. (이 집합을 "진열대"·"shelf"·"풀" 같은 별도 공간명으로 부르지 않는다 — 단지 Newsletters DB의 `unprinted` 레코드 집합일 뿐이다.)
- **프린트 시점에 매칭되는 미프린트 Newsletter 레코드가 없으면 즉석 생성으로 lazy 보충**한다. 그 시점에 **2개를 생성**한다 — 1개는 당일 출력용으로 바로 소비, 다른 1개는 Newsletters DB에 미프린트 상태로 남긴다.
- 스케줄된 Worker는 이 축을 해치지 않는 범위에서만 사용. 기존 T−2h/T−30m Worker의 처분(제거 / 보조 역할로 재배치 / 유지)은 **designer 결정**.

### 반드시 들어가는 것 (IN)

- **키워드 관리**
  - 사용자가 키워드를 등록하면 Notion Keywords DB에 반영됨.
  - 키워드 입력이 주제 생성의 1차 트리거 (입력 시 자동으로 주제 생성 경로가 돌 수 있음 — 방식은 designer).
  - 키워드 예시: eBPF, XDP, DPDK, DPU, Geneve vs VXLAN Deep Dive, NUMA & CPU Affinity Optimization, GRO/GSO/TSO, ndo_features_check.

- **주제 관리 (3가지 생성 경로)**
  1. 키워드 등록 시 자동 생성 (Claude 호출). **생성 개수는 Claude가 판단** — 사용자가 수치를 지정하지 않음.
  2. 수동 주제 생성 버튼 (UI에서 사용자가 명시적으로 트리거).
  3. **사용자 직접 작성** (사용자가 원하는 주제 텍스트를 직접 입력해 Topics DB에 등록).
  - 3가지 경로 모두 MVP IN. 구체적 UI/상호작용은 designer 설계.
  - **과거 주제 회피 정책**: 자동·수동 생성 시 Claude는 "**이미 프린트된 주제**"와만 중복을 피한다. 아직 프린트 안 된 pending 주제와는 중복되어도 무방 (미프린트 Newsletter 레코드로 쌓아둘 수 있으니 OK).
  - **주제 생성 시 태그는 `모든주제` 단독으로 기본 부여된다.** Claude는 태그를 자동 추천·자동 분류하지 않는다. 사용자가 이후 수동으로 추가 태그를 붙이고 뗀다 (자동 생성 경로·수동 생성 경로·직접 작성 경로 전부 동일 규칙).

- **태그 시스템 (1급 개념, 전면 수동)**
  - 키워드·주제·뉴스레터가 공통으로 가지는 분류 축. 요일별 프린트 설정의 매칭 기준도 태그.
  - **주제 1건에 태그는 여러 개 붙을 수 있다** (예: 한 주제가 IT + 경제).
  - **Claude는 태그 시스템에 관여하지 않는다** — 자동 추천·자동 분류 모두 없음. 태그 생성과 Topic에의 태그 부여는 **전부 사용자 수동**.
  - **기본 태그 `모든주제`**: 모든 Topic에 기본으로 부여된다. 사용자는 이 태그를 **수정·삭제할 수 없다** (invariant). 다른 태그는 사용자가 수동으로 붙이고 뗀다.
  - 사용자가 새 태그를 만들 때는 UI 상에서 직접 생성한다 (Claude 동의 프롬프트 같은 자동화 흐름 없음).
  - 태그 스키마·정규화·편집 UI·수동 부여 플로우는 designer 설계.

- **뉴스레터 관리**
  - 뉴스레터 레코드 1건 = 생성된 아티클 1건. Newsletters DB에 `unprinted` / `printed` 상태로 존재하며, **아직 프린트되지 않은 레코드 집합**에는 별도 공간명(진열대·shelf·풀)을 부여하지 않는다.
  - **생성 경로**:
    1. **수동 생성 버튼** — 사용자가 주제를 골라 즉시 뉴스레터 1건을 만들고 Newsletters DB에 미프린트 상태로 추가.
    2. **프린트 시점 lazy 보충** — 슬롯에 매칭되는 미프린트 Newsletter 레코드가 없으면 그 시점에 **2개 생성** (1 사용 + 1 미프린트로 잔류).
  - **1개 뉴스레터 = 주제 N개를 엮어 만든다.** N은 **Claude가 자동으로 결정**한다. 사용자는 **슬롯 단위의 "장수(페이지 수)"만 지정**하고, 그 장수에 맞게 Claude가 N과 각 주제 분량을 자동 배분한다.
  - **사용된 주제는 소비(consumed) 처리**되어 이후 뉴스레터 생성에 다시 뽑히지 않는다.
  - 뉴스레터 태그는 구성 주제의 태그에서 파생 (정확한 도출 규칙은 designer 설계 — 합집합/슬롯 태그 기준 등).

- **요일별 프린트 설정 — "슬롯 묶음" 구조 (MVP는 단일 슬롯 1경로)**
  - 각 요일은 **(프린트 시각, 슬롯 묶음)** 으로 정의된다.
  - **슬롯 = (태그, 장수)**.
  - **MVP 슬롯 구성은 `[(모든주제, N장)]` 단일 슬롯**. 서로 다른 태그 여러 개를 같은 요일에 묶는 다중 슬롯 구성, 그리고 그때 발생하는 슬롯 간 격리·순서·매칭 검증은 **공개 배포 마일스톤으로 이월**한다.
  - 구조적으로는 다중 슬롯을 수용할 수 있어도 (아래 규칙 — 요일 × 태그 유일성, 순서 무관 등), MVP 판정 경로와 테스트는 단일 슬롯만 커버한다.
  - **같은 요일 안에서 동일 태그 슬롯은 중복 불가** (요일 × 태그 = 유일).
  - 한 요일의 슬롯은 프린트 시점에 Newsletters DB의 미프린트 레코드와 매칭되며, 매칭된 뉴스레터가 태그와 장수 조건을 만족해야 한다.
  - **한 슬롯에 후보 뉴스레터가 여러 개면 Claude가 그 날의 출력으로 추천**.
  - **슬롯 출력 순서는 무관** (같은 요일 내 슬롯 간 정해진 순서 없음 — 다중 슬롯 이월 이후 유효).
  - **요일별 수동 on/off 토글**: 한 번 OFF로 끄면 사용자가 다시 켤 때까지 그 요일은 계속 OFF (자동 복구 없음). OFF된 요일은 시각 도달해도 아무 것도 출력 안 함.

- **프린트 (슬롯 매칭 + lazy 보충)**
  - 요일의 프린트 시각이 오면 그 요일의 슬롯(MVP는 `[(모든주제, N장)]` 1개)을 처리한다.
  - 슬롯 처리 규칙:
    1. 해당 태그·장수에 맞는 Newsletters DB의 미프린트 레코드 후보를 찾는다.
    2. 후보가 1개 이상이면 Claude가 추천한 1건을 출력. 그 뉴스레터는 `printed` 처리.
    3. 후보가 0개면 그 시점에 **2개 생성** (1개 출력, 1개 미프린트로 잔류).
    4. 주제도 비어 생성 자체가 불가하면 그 슬롯은 건너뛰고 사용자에게 알림.
  - 슬롯 간 실패 격리는 다중 슬롯 이월 이후 유효. MVP에서는 단일 슬롯 실패 = 당일 출력 실패로 귀결.
  - 출력 경로는 IPP (Canon G3910).

- **Notion 3개 DB 초기 설정** (Keywords/Topics/Newsletters) + 태그/장수/요일설정 저장 위치 (designer 결정).

- **설정 UI**: API 키, 프린터 IP, 요일별 슬롯 묶음 편집 (시각·태그·장수·on/off 토글).

### 지금은 빠지는 것 (OUT)

- **CleanupWorker (00:00 resolved 키워드 정리)** — MVP 판정 대상 아님. 코드 유지 여부는 designer 판단.
- 공개 배포용 온보딩/에러 UX/시크릿 위생 정리 — 다음 마일스톤 (`docs/plans/release-hardening.md` 별도 트랙).

- 뉴스레터 재발행, 키워드 가중치, 여러 사용자·프린터 등 고급 기능.

## 4. MVP 완성 판정 기준 (Acceptance)

**단일 판정 기준: 사용자가 1일 E2E를 한 번 끝까지 성공시킨다.**

아래 순서를 사용자 본인이 한 번 완주하면 MVP 완성.

1. 앱 최초 실행 → Notion 연동 초기 설정 (Keywords/Topics/Newsletters DB 자동 생성).
2. 설정 화면: Claude API 키 · 프린터 IP · 당일 요일의 프린트 시각 입력.
3. 키워드 등록 (예: eBPF, XDP, DPDK, DPU, Geneve vs VXLAN Deep Dive, NUMA & CPU Affinity Optimization, GRO/GSO/TSO, ndo_features_check 중 사용자가 고른 것들).
4. 키워드 자동 주제 생성 확인 (또는 수동 생성 버튼 / 직접 작성 중 하나 이상의 경로 동작). 생성된 Topic에는 `모든주제` 태그가 기본 부여되어 있다.
5. 사용자가 Topic에 수동으로 태그를 추가·제거할 수 있음을 확인 (자동 추천/자동 분류 플로우는 MVP에 존재하지 않는다).
6. 요일별 프린트 설정에 **단일 슬롯** 등록 (예: 당일 요일 · 가까운 미래 시각 · `[(모든주제, 2장)]`).
7. 프린트 시점 도달 — 매칭 미프린트 Newsletter 레코드가 없을 경우 즉석 2개 생성 → 1개 출력, 1개 미프린트로 잔류.
8. Canon G3910에서 종이 출력 확인.

**핵심 불안 지점**: 현재까지 사용자는 이 1~8번을 한 번도 끝까지 돌려본 적이 없다. MVP 판정은 "코드가 돈다"가 아니라 **"사용자가 실제로 종이를 받아봤다"** 다.

## 5. 상태 전이 규칙 (MVP 범위)

간단히 정리 — 세부 구현(필드명·저장 위치)은 designer.

- **키워드**: `활성` / `resolved`(주제 생성에 사용됨). MVP에서 resolved 자체는 프린트 동작과 직접 연관 없음.
- **주제 (Topic)**: `pending` (아직 뉴스레터에 쓰이지 않음) / `consumed` (뉴스레터에 엮여서 사용됨, 재사용 불가).
- **뉴스레터 (Newsletter)**: `unprinted` (Newsletters DB에 미프린트로 잔류) / `printed` (출력 완료, 과거 주제 회피의 기준이 되는 상태).

## 6. 엣지 케이스 / 실패 모드 (MVP 정책)

| 상황 | MVP 정책 |
| --- | --- |
| 프린트 시각에 프린트 실패 (프린터 off, 네트워크 단절 등) | **당일 1회 재시도**. 그래도 실패면 포기하고 사용자에게 알림. 재시도 상세는 designer. |
| pending 키워드가 0개라 자동 주제 생성 트리거가 돌 수 없음 | **기본 스킵**. 설정에 "과거 주제 기반 자동 생성 활성화" 토글을 두고 기본값은 OFF (designer 구현). |
| 슬롯 매칭 실패 (태그 후보 주제/뉴스레터 모두 0개) | MVP 단일 슬롯 기준: 그 슬롯을 스킵하고 사용자에게 알림. (다중 슬롯 이월 후에는 "가능한 슬롯만 출력, 실패 슬롯은 알림" 규칙으로 확장 — 요일 전체를 무효화하지 않음.) |
| 뉴스레터 생성 실패 (Claude API 오류 등) | **알림만 표시**, 해당 슬롯 스킵. 자동 재시도 없음. |
| 요일 OFF 토글이 켜진 날 | 해당 요일의 모든 슬롯 처리 **완전 스킵**. 사용자가 다시 켤 때까지 유지 (자동 ON 복구 없음). |
| Notion DB 초기 설정 실패 (토큰/권한) | 사용자가 원인을 알 수 있는 수준의 UI 피드백. 상세 UX는 designer. |
| 사용자가 앱 강제 종료 / 도즈 모드 | WorkManager 신뢰성은 designer 검증 대상. 단 MVP는 "1일 E2E 성공" 기준이므로 장기 신뢰성은 MVP 판정 범위 밖. |

위 정책의 **구체적 처리 방식** (알림 문구, 재시도 파라미터, 토글 UI 위치 등)은 designer 설계 영역.

## 7. 비범위 (NOT in scope)

- CleanupWorker를 MVP 판정에 포함시키는 것.
- 장기 신뢰성 (며칠 연속 무인 동작, 도즈 회피 심화).
- 키워드 가중치·주제 편집·뉴스레터 재발행 등 고급 기능.
- 여러 사용자·여러 프린터.
- 자동화 테스트 스위트.
- 공개 배포 하드닝.

## 8. 디자이너 핸드오프

designer는 아래 **8개 항목**을 독립 플랜으로 쪼개 설계한다. 새 기능 추가가 아니라 **MVP의 1일 E2E가 성립하도록 파이프라인을 재정의·검증**하는 것이 목적.

각 플랜은 공통 3단 구조: (a) 현재 코드 맥락 파악 → (b) 최소 수동 E2E 시나리오 + 사용자 체크포인트 → (c) 깨진 곳 수리 범위.

1. **키워드 등록 → 자동 주제 생성 E2E 검증**
   - 등록 트리거·Claude 호출 경로 재설계. **생성 개수는 Claude가 판단**.
   - 기존 `DailyTopicWorker`의 처분(유지/재배치/제거) 여기서 결정.
   - 과거 주제 회피는 **프린트된 주제만 기준** — pending은 중복 허용.

2. **수동 주제 생성 + 사용자 직접 작성 경로**
   - 수동 주제 생성 버튼 UI + 동작 흐름.
   - **사용자 직접 작성 입력 폼** (주제 텍스트 입력 → Topics DB 등록).
   - 생성된 Topic은 모두 `모든주제` 태그가 기본 부여된 상태로 저장된다. Claude는 태그 자동 부여에 관여하지 않으며, 사용자가 이후 수동으로 추가 태그를 붙이고 뗀다.

3. **태그 시스템 (전면 수동, Claude 비관여)**
   - 주제 1건에 태그 **여러 개** 부여 가능한 데이터 모델.
   - **태그 생성·Topic에의 태그 부여는 전부 사용자 수동.** Claude 자동 추천·자동 분류는 없다.
   - **`모든주제` 태그 invariant**: 모든 Topic에 기본 부여, 수정·삭제 불가. (이전 이름 `자유주제`는 폐기 — rename 필요.)
   - 사용자가 새 태그를 만드는 수동 생성 UI·기존 태그 부여 UI·편집/삭제 UX.
   - 태그 정규화·저장 위치. (자동 동의 프롬프트는 폐기.)

4. **뉴스레터 lazy 보충 모델 설계**
   - Newsletters DB를 `unprinted` / `printed` 상태로 운영하는 상태 전이 (미프린트 레코드 집합에 별도 공간명을 부여하지 않는다).
   - **수동 뉴스레터 생성 버튼**.
   - **프린트 시점 lazy 보충 — 매칭 미프린트 Newsletter 레코드가 없으면 2개 생성 (1 사용, 1 미프린트로 잔류)** 정책 구현.
   - 기존 `NewsletterWorker` / `T−30m` 스케줄의 처분 결정.
   - 사용된 주제 consumed 처리 규칙.

5. **다중 주제 엮기 (N=auto)**
   - 1 뉴스레터 = 주제 N개 엮음. **N과 각 주제 분량은 Claude 자동** 결정.
   - 사용자는 **슬롯의 장수(페이지 수)만 지정**, Claude가 그 분량에 맞게 N을 정한다.
   - 생성 실패/분량 초과 시 정책.

6. **요일별 프린트 설정 UI (슬롯 묶음, MVP는 단일 슬롯)**
   - 요일당 **(시각, [(태그, 장수), ...])** 편집 UI. **MVP 기본값은 `[(모든주제, N장)]` 단일 슬롯.**
   - **요일 × 태그 유일성 제약**은 데이터 모델 레벨에서 강제 (다중 슬롯 이월 대비).
   - **요일별 on/off 수동 토글** (OFF 유지 규칙).
   - 스케줄러는 단일 슬롯 경로를 MVP로 검증. 다중 슬롯 확장은 공개 배포 마일스톤.

7. **IPP 프린트 E2E (Canon G3910)**
   - `PrintService`의 IPP 경로 실기 검증.
   - **MVP는 단일 슬롯 1회 출력 기준**. 한 요일 내 슬롯 여러 개 연속 출력 (순서 무관, 실패 격리)은 공개 배포 이월.
   - 한 슬롯 후보 여러 개면 **Claude 추천으로 1건 선정**.
   - 프린터 IP 입력 UI 포함.

8. **1일 E2E 리허설 플랜**
   - 위 1~7이 합류한 상태에서 사용자가 직접 돌리는 최종 리허설 시나리오.
   - 실패 시 어느 플랜으로 되돌아갈지 매핑.
   - **CleanupWorker는 리허설 대상 제외**.

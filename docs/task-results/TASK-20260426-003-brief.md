# Task Brief: tag-system 3·4·6단계 — DB 시드 + Repository 태그 read/write + 호출지 빈 리스트 임시 처리

Task ID: TASK-20260426-003
Status: active
Owner: planner → implementer
Plan: docs/plans/tag-system/

## Goal

`docs/plans/tag-system/03-checklist.md`의 **3·4·6단계**를 한 번에 마쳐 컴파일이 깨지지 않게 통합한다(4단계가 saveTopic/saveNewsletter 시그니처를 바꾸므로 6단계의 임시 빈-리스트 통과가 동시에 필요). 5단계(옵션 풀 헬퍼)는 핸드오프 #6으로 이월(사용자 확정). 7단계(수동 E2E)·8단계(상태 갱신)는 본 task 결과 보고에 포함.

라운드 1 결정 반영:
- 시드 태그 이름 `모든주제` (코드 상수 식별자 `FREE_TOPIC_TAG`는 변경하지 않음 — 별도 implementer task에서 rename 예정).
- Claude는 태그에 관여하지 않음 — 6단계의 service 호출 지점들도 태그를 emptyList()로 전달.
- `ensureFreeTopicTag()` invariant가 saveTopic 한 곳에서 강제 → emptyList() 전달도 안전(`모든주제`로 자동 보충).

## User-visible behavior

- 직접적인 UI 변화 없음(태그 부여 UI는 핸드오프 #2/#3 후속).
- `NotionSetupService.setupDatabases()`가 3개 DB 생성 시 각 DB의 properties에 `Tags multi_select` + 시드 옵션 `모든주제`(color=`gray`)를 등록 → 신규 setup 시 Notion에서 3개 DB 모두 `Tags` 컬럼 + 시드 노출.
- 디버그 진입점이나 수동 호출로 `KeywordRepository.addKeyword("테스트", "keyword", listOf("신규태그"))` → Keywords DB 페이지에 `Tags = [신규태그]`.
- `TopicRepository.saveTopic(..., tags = emptyList())` → invariant에 의해 Topics DB에 `Tags = [모든주제]` 단독 자동 부착.

## Scope

### 3단계 (NotionSetupService 시드)
- `service/NotionSetupService.kt`의 3개 DB 생성 호출 properties에 `"Tags"` multi_select 추가.
- 시드 옵션 1개: 이름 `"모든주제"`, color `"gray"`.
- 마이그레이션 분기 없음(본인 1명, 기존 데이터 폐기 가능 전제).

### 4단계 (Repository 태그 read/write)
- UI 모델 3종에 `tags: List<String>` 필드 추가:
  - `KeywordUiItem` (위치: `data/repository/KeywordRepository.kt` 또는 동급)
  - `TopicUiItem` (위치: `data/repository/TopicRepository.kt` 또는 동급)
  - `NewsletterUiItem` (위치: `data/repository/NewsletterRepository.kt` 또는 동급)
- `KeywordRepository`:
  - `addKeyword(text, type, tags: List<String>)` 시그니처 확장. 반환 타입은 변경하지 않음(현재 Unit 유지 — `KeywordUiItem` 반환 변경은 topic-generation-paths 4단계 영역).
  - `refreshKeywords` / `getPendingKeywords` 매핑에 `tags` 전파.
- `TopicRepository`:
  - `saveTopic(title, priorityType, sourceKeywordIds, tags: List<String>)` 시그니처 확장.
  - **내부에서 `ensureFreeTopicTag(tags)` 적용** 후 Notion multi_select 페이로드 구성. invariant 강제 지점은 여기 한 곳.
  - `getTodayTopics` 매핑에 `tags`. `getAllPastTopicTitles`는 시그니처 변경 없음(매핑만 안 깨지게 유지).
- `NewsletterRepository`:
  - `saveNewsletter(title, htmlContent, topicIds, pageCount, tags: List<String>)` 시그니처 확장.
  - `getNewsletters` 매핑에 `tags`.
  - **신규 메서드** `findUnprintedNewsletterByTag(tagName: String): NewsletterUiItem?` — Notion 쿼리 (`multi_select.contains` + `Status.equals("generated")`) 결과 첫 페이지의 첫 항목 반환(없으면 null).

### 6단계 (service 호출지 빌드 유지)

- `service/NewsletterGenerationService.kt` — `saveNewsletter(..., tags = emptyList())`로 호출.
- 각 호출 지점 위에 한 줄 TODO 주석:

  ```kotlin
  // TODO(round-1): 라운드 1 — Claude는 태그에 관여하지 않는다. 자동/수동/직접 작성 모든 경로의 default 태그는 `모든주제` 단독.
  //                현재는 saveTopic/saveNewsletter 내부의 ensureFreeTopicTag invariant가 보충. 추가 태그는 사용자 수동 부여(핸드오프 #2 영역).
  ```
- `service/TopicSelectionService.kt`는 topic-generation-paths plan에서 삭제 예정 — 본 task에서는 컴파일 통과 목적으로만 emptyList() 전달.
- `ui/keyword/KeywordViewModel.kt`, `ui/keyword/KeywordScreen.kt`, `ui/topics/TopicsViewModel.kt`, `ui/newsletter/NewsletterViewModel.kt` 의 호출 지점도 동일 임시 처리 — 시그니처 매개변수 추가에 따른 컴파일 유지가 목적.

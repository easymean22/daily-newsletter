# Task Brief: Notion 초기 설정 UI 트리거 + 크래시 안전망

Task ID: TASK-20260426-006
Status: active
Owner: planner → implementer

## Context (왜 긴급)

사용자가 Scope A 1차 시도에서 키워드 저장 시 fatal crash 발생:
```
java.lang.IllegalStateException: Keywords DB가 초기화되지 않았습니다
  at KeywordRepository.getDbId(KeywordRepository.kt:29)
  ...
  Dispatchers.Main.immediate
```

근본 원인:
1. **`NotionSetupService.setupDatabases()` 호출 사이트가 코드베이스에 0건** — Settings에 트리거 버튼이 없어 Notion에 3개 DB가 만들어지지 않음. 그러므로 `KEY_KEYWORDS_DB_ID`는 항상 null이고, Repository의 `getDbId()`가 ISE를 던짐.
2. **try/catch가 있어도 fatal로 도달** — `KeywordViewModel.addKeyword`의 외층 try/catch가 있는데 스택에 ViewModel 프레임이 안 보이고 Dispatchers.Main.immediate에 그대로 전달. 빌드 캐시·suspend 예외 전파 quirk·다른 ViewModel의 unhandled launch 등 가능성. 안전망이 추가로 필요.

## Goal

(A) **Settings 화면에 "Notion DB 자동 생성" 트리거 추가** — 사용자가 명시적으로 누르면 `NotionSetupService.setupDatabases()` 실행. 진행 상태·성공·실패 모두 UI 피드백.

(B) **크래시 안전망 추가** — 4개 ViewModel(`Keyword/Topics/Newsletter/Settings`)에 공통 `CoroutineExceptionHandler` 적용 + UI에서 `error` 상태를 스낵바로 표시. 향후 unhandled exception이 fatal로 가지 않도록.

(C) **`KeywordViewModel.addKeyword`의 try/catch 우회 여부 추적 메모** — verifier가 회귀 테스트로 확인.

## User-visible behavior

### (A) Settings 트리거
- Settings 화면에서 Notion API Key + Parent Page ID 입력 후 **"Notion DB 자동 생성" 버튼**(또는 "초기 설정 시작") 노출.
- 버튼은 Notion API Key + Parent Page ID 둘 다 비어있지 않을 때만 enabled. 비어있으면 disabled + 안내 텍스트("Notion 토큰과 상위 페이지 ID를 입력해주세요").
- 탭 → 진행 인디케이터 표시 + 다른 입력 비활성화 → `setupDatabases()` 실행 → 성공 시 "Notion DB 3개가 생성되었습니다" 스낵바 + 버튼이 "재설정 필요" 안내로 바뀌거나 비활성화. 실패 시 에러 메시지 스낵바("DB 생성 실패: <message>").
- 이미 setup이 완료된 경우(`KEY_KEYWORDS_DB_ID != null`) `setupDatabases()`는 즉시 return하므로(현재 동작), 버튼은 "이미 생성됨" 같은 disabled 상태로 표시.

### (B) 크래시 안전망 + 에러 표시
- `KeywordScreen`이 `state.error`를 collect하여 비어있지 않으면 스낵바로 표시 후 `clearError()` 호출(현재 `state.error`가 UI에 표시되지 않아 silent 실패 발생 중).
- `Topics/Newsletter/Settings` ViewModel도 동일 패턴(이미 error 상태가 있다면 UI 노출, 없다면 추가).
- 각 `@HiltViewModel` 클래스에 `private val exceptionHandler = CoroutineExceptionHandler { _, e -> _uiState.update { it.copy(error = e.message ?: "...") } }`를 두고, `viewModelScope.launch(exceptionHandler) { ... }` 패턴으로 unhandled exception을 흡수.

## Scope

### Part A — Settings 트리거
- `app/src/main/java/com/dailynewsletter/ui/settings/SettingsViewModel.kt`:
  - `NotionSetupService` 주입.
  - `SettingsUiState`에 `isSetupRunning: Boolean`, `setupResult: SetupResult?` 추가 (sealed class: `Idle`, `Running`, `Success`, `Failed(message)`).
  - `runSetup()` 메서드 신설: viewModelScope.launch(exceptionHandler) { setupRunning = true → setupDatabases() → success/fail 갱신 → setupRunning = false }.
- `app/src/main/java/com/dailynewsletter/ui/settings/SettingsScreen.kt`:
  - "Notion DB 자동 생성" 버튼 추가 — 위치는 Notion 토큰·페이지 ID 입력 필드 근처.
  - enabled 조건: Notion API Key + Parent Page ID 둘 다 not blank + setupResult != Running.
  - 이미 `KEY_KEYWORDS_DB_ID`가 set 됐으면 버튼 disabled + "이미 생성됨 (KEY_KEYWORDS_DB_ID 존재)" 안내.
  - 진행 중 ProgressIndicator + 결과 스낵바.
- `app/src/main/java/com/dailynewsletter/data/repository/SettingsRepository.kt` 변경 X(이미 `getKeywordsDbId` 존재).

### Part B — 크래시 안전망
- 공통 helper 신설(또는 각 ViewModel에 인라인): `viewModelScope.launchSafe(...)` 또는 `viewModelScope.launch(exceptionHandler)` 패턴.
  - 권장: `app/src/main/java/com/dailynewsletter/ui/common/ViewModelExtensions.kt` 신설. `fun ViewModel.exceptionHandler(onError: (String) -> Unit) = CoroutineExceptionHandler { _, e -> onError(e.message ?: "Unknown error") }`.
- 4개 ViewModel(`Keyword/Topics/Newsletter/Settings`) 각 클래스에:
  - `_uiState`에 `error: String?` 필드(없으면 추가).
  - `exceptionHandler`를 만들어서 `error` 갱신.
  - 모든 `viewModelScope.launch { ... }` 를 `viewModelScope.launch(exceptionHandler) { ... }` 로 변경.
  - `clearError()` 메서드 추가.
- 각 Screen에서 LaunchedEffect로 `error`를 collect → `snackbarHostState.showSnackbar(error)` → `viewModel.clearError()`.
- `KeywordScreen`은 이미 `snackbarHostState`가 있으므로 `LaunchedEffect(state.error)` 한 블록만 추가.

### Part C — 추적 메모
- `docs/task-results/TASK-20260426-006-implementer-result.md`에 verifier 노트로 다음 항목 포함:
  - 빌드 깨끗하게 다시 (gradle clean → assembleDebug) 후 단말 재설치 권장.
  - try/catch가 정상 동작하는지 회귀 테스트: Notion DB 미생성 상태에서 키워드 저장 → 크래시 없고 스낵바로 에러 표시되는지.
  - try/catch가 우회된다면 다른 launch 사이트(특히 `KeywordViewModel.toggleResolved`, `deleteKeyword`)도 안전망 적용 후 회귀 확인.
